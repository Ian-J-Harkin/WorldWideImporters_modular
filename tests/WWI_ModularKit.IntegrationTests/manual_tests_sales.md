To verify that your Sales Subsystem is production-ready, perform these manual tests using a tool like Postman, Insomnia, or the Swagger UI (usually at http://localhost:[port]/swagger).

These tests confirm that your "Guardrails" are active and that the Transactional Outbox is functioning across schemas.

🛠️ Manual Verification Test Suite: Sales Subsystem
Test 1: Multi-Tenancy Header Enforcement
Action: Submit a valid POST request to /api/v1/orders but omit the X-Tenant-Id header.

Expected Result: * Status Code: 400 Bad Request.

Response Body: Should contain an error message: "Tenant ID is required."

Verification: This confirms the HttpTenantProvider guardrail is preventing "orphan" data.

Test 2: Schema Isolation & Auto-Stamping
Action: Submit a valid POST request to /api/v1/orders with header X-Tenant-Id: 3f2b1c4d-5e6f-7a8b-9c0d-1e2f3a4b5c6d.

Expected Result: * Status Code: 201 Created.

Database Check: Open your Postgres tool (pgAdmin/DBeaver) and run:
SELECT * FROM sales."Orders";

Verification: 1.  The table must exist in the sales schema.
2.  The TenantId column must be automatically populated with 3f2b1c4d... even though you didn't include it in the JSON body.

Test 3: Global Query Filter (Isolation)
Action: 1.  Submit an Order for Tenant A (X-Tenant-Id: AAAAA...).
2.  Submit a GET request for all orders using Tenant B (X-Tenant-Id: BBBBB...).

Expected Result: * Response Body: Should return an empty list [] or only orders belonging to Tenant B.

Verification: This confirms the BaseDbContext expression tree is dynamically filtering data at the database level.

Test 4: Transactional Outbox Integrity
Action: Perform a successful POST /api/v1/orders.

Expected Result: * Database Check: Immediately run:
SELECT * FROM sales."OutboxMessages";

Verification: You should see a new row containing the OrderCreatedIntegrationEvent in JSON format.

Success Condition: If you stop the RabbitMQ container before this test, the row should stay in the table. If RabbitMQ is running, the row may disappear quickly (processed), but you can verify it reached the bus via the RabbitMQ Management UI (localhost:15672).

Test 5: Validation Logic (FluentValidation)
Action: Submit a POST request with a negative quantity or an empty list of items.

Expected Result: * Status Code: 400 Bad Request.

Verification: The response should list specific validation errors generated by CreateOrderValidator.cs.
