# 🧪 Manual Verification Test Suite: Sales Subsystem

To verify that your Sales Subsystem is production-ready, perform these manual tests using a tool like Postman, Insomnia, or the Swagger UI (usually at http://localhost:5000/swagger).

These tests confirm that your "Guardrails" are active and that the Transactional Outbox is functioning across schemas.

---

## 🏗️ Technical Architecture: Hybrid Model
> [!IMPORTANT]
> This project uses a **Hybrid Development Environment**. To test the full flow, you must have **three distinct layers** running simultaneously:
>
> 1. **Infrastructure (Docker)**: Postgres and RabbitMQ must be running in Docker.
> 2. **Backend API (Local)**: The .NET Host project must be running locally via `dotnet run`.
> 3. **Frontend UI (Local)**: The Angular client must be running locally via `npm start`.

---

## 🛠️ Phase 0: Pre-flight Diagnostics (The "Three-Terminal" Setup)
Before starting the tests, ensure your local environment is correctly configured. You will need at least **three terminal windows** open.

### **Terminal 1: Infrastructure (Docker)**
1. **Check Backend Services**:
   Run the following to ensure Postgres and RabbitMQ are running:
   ```powershell
   docker ps
   ```
   *Verification*: You should see `postgres:16-alpine` and `rabbitmq:3-management` with status `Up`.

### **Terminal 2: Backend API (.NET)**
1. **Start the API Host**:
   ```powershell
   cd c:/Github/WWI_ModularKit
   dotnet run --project src/Host
   ```
   *Verification*: Wait for the terminal to say `Now listening on: http://localhost:5128`.

### **Terminal 3: Frontend UI (Angular)**
1. **Start the Client**:
   ```powershell
   cd c:/Github/WWI_ModularKit/client
   npm start
   ```
   *Verification*: Wait for the terminal to say `✔ Compiled successfully.` and navigate to `http://localhost:4200`.

## 🚀 Manual Tests

### Test 1: Multi-Tenancy Header Enforcement
**Action**: Submit a valid POST request to `/api/sales/orders` but **omit** the `X-Tenant-Id` header.

**Expected Result**:
* **Status Code**: 400 Bad Request.
* **Response Body**: Should contain an error message: `"Tenant ID is required."`

**Verification**: This confirms the `HttpTenantProvider` guardrail is preventing "orphan" data.

---

### Test 2: Schema Isolation & Auto-Stamping
**Action**: Submit a valid POST request to `/api/sales/orders` with header `X-Tenant-Id: 3f2b1c4d-5e6f-7a8b-9c0d-1e2f3a4b5c6d`.
*Note: You must first seed a Customer in the database (see Seed snippet below) or the request will fail with a Foreign Key violation.*

**Expected Result**:
* **Status Code**: 201 Created.

**Database Check**: Open your Postgres tool and run:
   ```sql
   SELECT * FROM sales."Orders";
   ```

**Verification**:
1. The table **must** exist in the `sales` schema.
2. The `TenantId` column must be **automatically populated** with `3f2b1c4d...` even though you didn't include it in the JSON body.

---

### Test 3: Global Query Filter (Isolation)
**Action**: 
1. Submit an Order for **Tenant A** (`X-Tenant-Id: 11111111-1111-1111-1111-111111111111`).
2. Submit an Order for **Tenant B** (`X-Tenant-Id: 22222222-2222-2222-2222-222222222222`).
3. Submit a **GET** request to `/api/sales/orders` using the header for **Tenant A**.
4. Submit a **GET** request to `/api/sales/orders` using the header for **Tenant B**.

**Expected Result**:
* Request 3 (Tenant A) should return **only** Tenant A's orders.
* Request 4 (Tenant B) should return **only** Tenant B's orders.

**Verification**: This confirms the `BaseDbContext` global query filter is automatically isolating data at the database level.

---

### Test 4: Transactional Outbox Integrity
**Action**: Perform a successful POST `/api/sales/orders`.

**Expected Result**:
* **Database Check**: Immediately run:
  ```sql
  SELECT * FROM sales."OutboxMessages";
  ```

**Verification**: You should see a new row containing the `OrderCreatedIntegrationEvent` in JSON format.

*Success Condition*: If you stop the RabbitMQ container before this test, the row should stay in the table. If RabbitMQ is running, the row may disappear quickly (processed), but you can verify it reached the bus via the RabbitMQ Management UI (`localhost:15672`).

---

### Test 5: Validation Logic (FluentValidation)
**Action**: Submit a POST request with a negative quantity or an empty list of items.

**Expected Result**:
* **Status Code**: 400 Bad Request.

**Verification**: The response should list specific validation errors generated by `CreateOrderValidator.cs`.

## 🎨 Phase 1: Frontend (Angular) UI Verification
Once the backend is confirmed healthy, verify the production guardrails from the user interface.

1. **Start the Frontend**:
   ```powershell
   cd client
   npm start
   ```
   *Verification*: Navigate to `http://localhost:4200`.

2. **Test 6: UI Multi-Tenancy Guardrail** (Production Guardrail 1)
   - **Action**: Without clicking "Set Test Tenant", enter a Customer ID and click **Submit Order**.
   - **Expected Result**: 
     - The request must **not** reach the backend.
     - Check the Browser Console (F12): You should see an error: `"Tenant ID is required for all requests."`
   - **Verification**: This confirms the `tenantInterceptor` is enforcing isolation at the client level.

3. **Test 7: Signal-Based State Sync**
   - **Action**: Click the **"Quick Add: USB Ribbon"** button.
   - **Expected Result**: 
     - The "Order Items" list updates instantly.
     - The "Total Amount" at the bottom updates instantly to `$75.00`.
   - **Verification**: This confirms the Angular **Signals** and **computed** values are correctly wired.

4. **Test 8: End-to-End Success**
   - **Action**: 
     1. Click **"Fix: Set Test Tenant"** in the top bar.
     2. Ensure the status shows `Tenant: 8db1620a...` (Green indicator).
     3. Click **Submit Order**.
   - **Expected Result**: 
     - Browser Console shows: `Order created: [GUID]`.
     - Check Database: `SELECT * FROM sales."Orders"` should show the new record with the correct tenant.
